# План внедрения HaloTrack на fenek-sport.cz

Документ для разработчика сайта — что именно будет установлено, как это работает и как откатить в случае проблем.

---

## Коротко

Устанавливается серверный трекинг заказов: при смене статуса заказа на «оплачен» сайт сам, автоматически, отправляет данные о заказе на внешний сервис (HaloTrack), который передаёт их дальше в Google Analytics/Google Ads. Существующий трекинг (GTM, Google Ads тег, Yandex Metrika, Seznam retargeting в `success.twig`) не убирается и не меняется — новая система работает параллельно.

**Ядро — таблица oc_order_halotrack и два обработчика событий OpenCart**, зарегистрированные через штатный механизм Events (`oc_event`), без модификации базовых файлов ядра (`checkout/order.php` и т.д.).

---

## 1. Что добавляется (новые файлы, ничего не перезаписывается)

```
catalog/controller/extension/module/halotrack.php
catalog/model/extension/module/halotrack.php
admin/controller/extension/module/halotrack.php
admin/view/template/extension/module/halotrack.twig
admin/language/en-gb/extension/module/halotrack.php
admin/language/cs-cz/extension/module/halotrack.php
```

Все пути новые — ни один существующий файл ими не затрагивается. До установки через админку это просто мёртвый код: OpenCart не выполняет файлы, на которые нет роутинга.

## 2. Что изменяется в существующих файлах (одна правка)

`catalog/view/theme/esport/template/common/header.twig` — перед `</head>` добавляется одна строка:

```html
<script src="https://<поддомен>.fenek-sport.cz/t.js" async></script>
```

Чистый HTML, без Twig-конструкций. Загружается асинхронно, не блокирует рендер страницы (но `async` не гарантирует, что скрипт успеет выполниться раньше JS чекаута — это отдельный, независимый от рендера момент).

**Поддомен здесь не опционален.** Пока трекинг работает на общем адресе (`*.vercel.app`), кука `_halo` физически не установится как first-party — OpenCart её просто не увидит, и `session_id` в `oc_order_halotrack` всегда будет пустым. Без настроенного CNAME на поддомен тестовый заказ подтвердит только доставку данных о заказе, но ничего не скажет об атрибуции — это два независимых результата теста, не один.

## 3. Изменения в базе данных

Новая таблица (существующие таблицы не трогаются). Префикс `oc_` подтверждён для этой конкретной инсталляции (`config.php`); в коде расширения он не захардкожен — используется `DB_PREFIX` динамически, ниже — для наглядности:

```sql
CREATE TABLE IF NOT EXISTS oc_order_halotrack (
  order_id INT(11) NOT NULL,
  session_id VARCHAR(64) NULL,
  customer_ip VARCHAR(45) NULL,
  forwarded_at DATETIME NULL,
  PRIMARY KEY (order_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8;
```

Плюс через штатный API настроек OpenCart (`oc_setting`, код `module_halotrack`) сохраняются: URL сервиса, секретный ключ для подписи запросов, список статусов заказа, которые считаются «оплачено».

## 4. Как это подключается к OpenCart (механизм)

OpenCart с версии 2.3+ имеет встроенную систему событий: любой вызов метода модели автоматически генерирует событие `model/<путь>/<метод>/after`, которое может перехватить зарегистрированный обработчик — без изменения самого метода. Это официальный, документированный механизм OpenCart, не хак. Проверено по исходникам `system/engine/loader.php` и `catalog/controller/startup/event.php` этой конкретной инсталляции.

Через `Admin → Extensions → Extensions → тип Modules → HaloTrack → Install` регистрируются два обработчика (в базе триггер хранится с префиксом `catalog/`, который `startup/event.php` сам отрезает перед регистрацией — без этого префикса обработчик молча не сработает ни разу):

| Триггер в базе (`oc_event.trigger`) | Когда срабатывает | Что делает наш код |
|---|---|---|
| `catalog/model/checkout/order/addOrder/after` | Создание нового заказа при оформлении | Читает cookie `_halo` (кто привёл покупателя), сохраняет вместе с order_id в `oc_order_halotrack` |
| `catalog/model/checkout/order/addOrderHistory/after` | Любая смена статуса заказа (колбэк банка, ручная смена в админке) | Если новый статус в списке «оплачено» — собирает данные заказа и одним HTTP-запросом отправляет их на HaloTrack |

Оба обработчика читают данные (заказ, товары) через штатные методы `catalog/model/checkout/order.php` — `getOrder()`, `getOrderProducts()`. Прямых SQL-запросов к таблицам заказов нет.

## 5. Исходящий запрос — что именно уходит и куда

Один POST-запрос на `https://<HaloTrack-домен>/api/webhook/order`:

- Содержимое: id заказа, сумма, валюта, email/телефон, состав заказа, id рекламной сессии, IP покупателя.
- Подпись: HMAC-SHA256 от секретного ключа — не статичный пароль в заголовке, а подпись с меткой времени, защищённая от повторного использования перехваченного запроса (5-минутное окно, совпадает со схемой на стороне HaloTrack).
- Таймаут: запрос синхронный, с ограничением в 2 секунды. Это не «не блокирует ответ» — это верхняя граница задержки: в худшем случае колбэк от банка или ответ администратору при смене статуса может занять до 2 секунд дольше обычного. Ошибка (недоступность, таймаут, неожиданные данные) перехватывается и не прерывает саму бизнес-операцию — заказ или смена статуса в любом случае завершится успешно.
- Если HaloTrack недоступен в момент запроса — повторной попытки нет. Ошибка попадает в лог, `forwarded_at` не проставляется, автоматического дозаказа не будет; заказ уйдёт только при следующей подходящей смене статуса, если она случится. Это та же граница, что и в уже работающем WooCommerce-плагине, не новая для этого проекта.
- `forwarded_at` означает только «HaloTrack принял и подтвердил вебхук» — не то, что Google Ads уже зарегистрировал конверсию. Это два разных шага, второй проверяется отдельно (см. п.9).

Входящих запросов НЕТ — сайт клиента ничего не принимает извне, только отправляет.

## 6. Порядок установки

1. Резервная копия: полный дамп файлов (FTP) + дамп базы данных (`System → Maintenance → Backup/Restore` или phpMyAdmin).
2. `php -l` (синтаксическая проверка) всех новых файлов локально — до заливки на сервер.
3. Заливка 6 новых файлов по FTP.
4. Правка `header.twig` (см. п.2) + сброс кэша темы, если сайт использует Theme Editor с копией в БД.
5. `Admin → Extensions → Extensions → Modules → HaloTrack → Install` — создаёт таблицу, регистрирует события.
6. Настройка модуля: URL, секретный ключ, список «оплаченных» статусов (Status оставить **Disabled** на этом шаге).
7. Проверка `php -l` ещё раз в проде (на всякий случай) + смотрим лог `system/storage/logs/error.log` на предмет ошибок при обычном заходе на сайт.
8. Включаем Status: Enabled.
9. Один тестовый заказ, оплата картой (чтобы пройти именно через колбэк платёжного шлюза GPWebPay) — проверяем, что строка появилась в `oc_order_halotrack` и `forwarded_at` заполнился.

## 7. Откат

| Ситуация | Действие | Время |
|---|---|---|
| Хотим просто приостановить | Admin → модуль HaloTrack → Status: Disabled | 10 секунд |
| Хотим убрать полностью | Admin → Extensions → Modules → Uninstall (удаляет события, таблицу намеренно не трогает) | 1 минута |
| Админка недоступна | SQL: `DELETE FROM oc_event WHERE code LIKE 'halotrack%'` | 1 минута |
| Совсем крайний случай | Удалить 6 файлов по FTP — OpenCart корректно обрабатывает отсутствие контроллера события (не роняет сайт), плюс вернуть `header.twig` из бэкапа п.1 | 2 минуты |

## 8. Что НЕ трогаем

- Ни одного файла ядра OpenCart.
- Ни одной существующей таблицы (только добавляем новую).
- SimpleCheckout, GPWebPay-интеграцию — код только читает результат их работы через события, не вмешивается в их логику.
- Существующий трекинг в `success.twig` (Yandex Metrika, GTM, Google Ads тег, Seznam, Google Customer Reviews) — работает как прежде, без изменений.

## 9. Известные открытые вопросы (сознательно не в объёме этого этапа)

- **Возвраты/отмены заказа.** Сейчас нет механизма сообщить в Google Ads/GA4, что оплаченный и переданный заказ позже отменён или возвращён — выручка останется учтённой. Не блокирует тестовый заказ, но нужно решить до масштабного использования данных для ставок.
- **Проверка на стороне Google.** Обычный `/mp/collect` endpoint Google Analytics может вернуть `2xx` даже для некорректного события — сам факт успешного ответа не доказывает, что событие принято к учёту. `forwarded_at` = «HaloTrack отправил», не «Google подтвердил». Перед тем как считать тест успешным, нужно дополнительно проверить событие через debug-инструмент Google (Measurement Protocol Validation Server) и убедиться, что `purchase` появился как key event в GA4 и импортирован в Google Ads.
- **Consent-баннер сайта не интегрирован.** Существующий `annytab_cookie_consent.js` не распознаётся `t.js` (тот знает только про CookieYes/Cookiebot/собственный localStorage-ключ) — сбор данных о сессии идёт независимо от выбора посетителя в баннере. Осознанно оставлено вне объёма этого этапа.
